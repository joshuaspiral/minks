\section*{1. Introduction and Research Question}
% copy pasted from proposal. will change soon.

Personal knowledge management (PKM) tools such as Obsidian(\cite{obsidian}) implement the Zettelkasten methodology(\cite {zettelkasten_method}), allowing users to organize their knowledge as a network of interlinked Markdown notes. A note in the network can link to other notes in the network through internal links called "WikiLinks", forming a graph where the notes are vertices and the links are edges. The benefit of such a system lies in its ability to provide users with insights into the relationship between ideas, and build a stronger understanding of their notes. However, many users (including ourselves), tend to under-link their notes, leading to a lackluster knowledge network. For example, one of the ways one of our group members uses Obsidian is through a "hub-and-spoke" structure, where minimal note vertices known as "hubs" form a directory with WikiLinks to other notes in the same topic (e.g. a hub named "CSC111.md" with WikiLinks to ("Tree.md", "Recursion.md", etc.)). This however, defeats the purpose of Obsidian, failing to capture the lateral connections notes can have in the network which actually make a knowledge graph useful. For instance, a link from "Trees.md" to "Abstract Syntax Trees (CS).md" to "Abstract Syntax Trees (Linguistics).md".

Whilst manually reviewing every note to add missing links is a valuable learning activity, it becomes highly impractical as a PKM system grows to hundreds or thousands of notes.

\textbf{"Can we use graph-based link prediction algorithms augmented with NLP-based semantic similarity to suggest meaningful missing connections in an Obsidian knowledge graph, and how do the algorithms' judgements compare with a user's own judgement of which notes should be linked?"}

\section*{2. Datasets}

To evaluate our prediction pipeline and avoid circular bias during parameter tuning, we used two distinct datasets:
\begin{itemize}
    \item \textbf{Vault A (Tuning Set):} A personal Obsidian vault belonging to one of the group members, consisting primarily of undergraduate course notes. This dataset was exclusively used in the \texttt{MinkPredictor.fit()} method to grid-search the optimal weighting between structural and semantic similarity scores.
    \item \textbf{Vault B (Test Set):} The open-source digital garden of Jacky Zhao (\url{jzhao.xyz}). This vault contains over 700 notes spanning computer science, philosophy, and other topics. It provides a denser and more diverse graph, allowing us to test our algorithm's out-of-domain generalizability. We utilized the raw \texttt{.md} plaintext and internal WikiLinks from this set.
\end{itemize}

\section*{3. Computational Overview}

\subsection*{Data Representation}
We model an Obsidian vault using a custom \texttt{KnowledgeGraph} class, representing the vault as an undirected graph $G = (V, E)$. Each Markdown note is parsed by \texttt{load\_graph.py} to strip YAML frontmatter and extract plaintext, becoming a \texttt{\_Note} vertex $v \in V$. WikiLinks between notes become edges $\{u, v\} \in E$.

\subsection*{Major Computations}
Our prediction pipeline, orchestrated by the \texttt{MinkPredictor} class, performs the following major computations:
\begin{enumerate}
    \item \textbf{Structural Similarity:} We manually implemented the Jaccard index and the Adamic-Adar index. Adamic-Adar assigns higher weight to shared neighbours with fewer connections, capturing the intuition that a rare mutual connection is stronger evidence for a missing link than a highly connected "hub" note.
    \item \textbf{Semantic Similarity:} To detect lateral connections between \textbf{structurally} isolated notes, we generate text embeddings. We built a manual \textbf{TF-IDF} (Term Frequency-Inverse Document Frequency) vectorizer from scratch to serve as a baseline, alongside an implementation using the \texttt{sentence-transformers} library for dense vector embeddings, which we use to calculate the Cosine Similarity between these vectors.
    \item \textbf{Weight Tuning \& Holdout Evaluation:} We compute a combined score $S = (w_{struct} \times \text{Structural}) + (w_{sem} \times \text{Semantic})$. We evaluate this using an edge holdout method: randomly removing 20\% of existing edges, predicting the top-$k$ missing links, and calculating \texttt{recall@k} and \texttt{precision@k}.
\end{enumerate}

\subsection*{Visualization and New Libraries}
We used three libraries to report our results interactively:
\begin{itemize}
    \item \textbf{NetworkX:} We convert our custom graph object into a \texttt{networkx.Graph} in \texttt{visualize.py} to use its \texttt{spring\_layout()} function, calculates 2D node coordinates based on structural edges.
    \item \textbf{Plotly:} We use \texttt{plotly.graph\_objects} to render the calculated coordinates into an interactive HTML visualization, visually distinguishing minks' predicted links (red, dashed) from existing WikiLinks (grey, solid).
    \item \textbf{Scikit-Learn (t-SNE):} To compare structural versus semantic clustering, we use \texttt{sklearn.manifold.TSNE} to reduce our high-dimensional text embeddings into a 2D space, plotted side-by-side with the NetworkX layout.
\end{itemize}

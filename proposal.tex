\section*{Problem Description and Research Question}

Personal knowledge management (PKM) tools such as Obsidian implement the Zettelkasten methodology, allowing users to organize their knowledge as a network of interlinked Markdown notes. A note in the network can link to other notes in the network through internal links called "WikiLinks", forming a graph where the notes are vertices and the links are edges. The benefit of such a system lies in its ability to provide users with insights into the relationship between ideas, and build a stronger understanding of their notes. However, many users (including ourselves), tend to under-link their notes, leading to a lackluster knowledge network. For example, one of the ways one of our group members uses Obsidian is through a "hub-and-spoke" structure, where minimal note vertices known as "hubs" form a directory with WikiLinks to other notes in the same topic (e.g. a hub named "CSC111.md" with WikiLinks to ("Tree.md", "Recursion.md", etc.). This however, defeats the purpose of Obsidian, failing to capture the lateral connections notes can have in the network which actually make a knowledge graph useful. For instance, a link from "Trees.md" to "Abstract Syntax Trees (CS).md" to "Abstract Syntax Trees (Linguistics).md".

Whilst manually reviewing every note to add missing links is a valuable learning activity, it becomes highly impractical as a PKM system grows to hundreds or thousands of notes.

\subsection{Research Question}
\textbf{"Can we use graph-based link prediction algorithms augmented with NLP-based semantic similarity to suggest meaningful missing connections in an Obsidian knowledge graph, and how do the algorithms' judgements compare with a user's own judgement of which notes should be linked?"}

\section*{Computational Plan}
\subsection*{Data Representation}
We will model an Obsidian vault as an undirected graph, $G = (V, E)$, where each Markdown note is a vertex $v \in V$ and each WikiLink between two notes is an edge $\{u, v\} \in E$. Despite graphs with WikiLinks syntactically being a directed graphs (a note titled "Graphs" would link to "Vertex" but not necessarily vice versa) we will represent the graph with an undirected graph, since for the purposes of link prediction, we care about the \textit{existence} of a relationship between two vertices, not its direction.

Each vertex $v$ will store the following instance attributes:
\begin{itemize}
    \item \texttt{note\_name}: the filename without the .md extension
    \item \texttt{note\_content}: the plaintext body after processing
\end{itemize}


We will use two datasets for our evaluation. The first is one of our group members' personal Obsidian vaults, used mainly in notetaking for course notes. The second is the open source \textit{digital garden} of Jacky Zhao (\url{jzhao.xyz}). Zhao's vault contains over 700 notes spanning computer science, philosophy and other topics, providing a denser and more diverse graph than our personal vault alone. Two datasets are used to allow us to test whether our pipeline generalizes beyond small, topic-narrow vaults.

Before constructing the graph, we will need to process the notes via a cleaning pipeline: stripping image embeddings, removing YAML frontmatter, removing tags, extracting all WikiLinkns, and extracting the remaining plaintex body for NLP-based semantic similarity.

\subsection{Computations}
We plan to implement link predictions in three stages. At each stage, we will tweak a similarity signal and evaluate how the predictions change. Hopefully we will be able to deduce how different similarity signals affect the judgements made by the algorithm. 

\textbf{Stage 1: Structural similarity (Jaccard)}
For each pair of non-adjacent vertices $(u,v)$, we can compute the Jaccard index over their neighbour sets:
$$
J(u,v) = \frac{|\Gamma(u) \cap \Gamma(v)|}{|\Gamma(u) \cup \Gamma(v)|}
$$
where $\Gamma(u)$ represents the set of neighbours of $u$. This captures the intuitive idea regarding structural similarity: if two notes share many common neighbours, then they are likely related.

\textbf{Stage 2: **Weighted** structural similarity (Adamic-Adar)}
The Adamic-Adar index is a weighted version of common-neighbour counting, assigning higher weight to shared neighbours with fewer connections: 

$$
A(u, v) = \sum_{w \in \Gamma(u) \cap \Gamma(v)} \frac{1}{\log |\Gamma(w)|}
$$

This captures the intuition that a rare mutual connection is more informative than a highly connected hub vertex \cite{adamic2003}. In a knowledge graph, a shared neighbour that is a highly specific concept (low degree) is stronger evidence for a missing link than a shared neighbour that is a broad hub note linking to everything.

:\subsection{Visualisation}

Interactive, graph-based visualisation of notes is a crucial component of PKM systems such as Obsidian. To plot user notes in an intuitive way, we plan to implement a pipeline that decouples the graph calculation algorithms from visualization. We plan to use the \textbf{NetworkX} and \textbf{Plotly} libraries. 


\subsubsection{NetworkX}
Our core graph algorithms will not use NetworkX. The identifaction of missing links and graph/vertex classes will be written entirely by us, resulting in a custom graph class that handles these computations. However, calculating a visually pleasing, untangled layout of the nodes in this graph involves complex physics-based computations. We plan to employ NetworkX's force-directed algorithms to accomplish this in an optimized fashion. 

Once our code has finished computing the link predictions, we plan to pipe our graph topology to a NetworkX \texttt{networkx.Graph} object. We will then pass this object to the \texttt{networkx.spring\_layout()} function. This function treats the graph as a physical system, treating edges as springs that pull notes together and vertices as magnets that repel each other. The function runs this physics simulation and outputs the calculated 2D coordinates of each node.
